# Scenario Tone — Midgame Survival Adventure

> 중반 이후 시나리오 제작 기준이다. 이 문서는 command catalog나 exercise bank를 대체하지 않는다. 모든 콘텐츠는 계속 `Vim command -> Exercise -> Scenario` 순서로 만든다.

## 결론

중반 톤은 **터미널 문제 해결 생존 어드벤처**로 간다.

플레이어는 전투를 하는 모험가가 아니라, 고장 난 시스템을 Vim 조작으로 복구하는 현장 오퍼레이터다. 생존감은 배경 압력이고, 미션의 본체는 항상 텍스트를 더 빠르고 정확하게 고치는 Vim command 학습이다.

## 톤 원칙

- 생존은 체력/전투/인벤토리가 아니라 로그, 설정, 통신, 임시 패치, 저장/폐기 판단에서 온다.
- 위기는 플레이어를 괴롭히는 장치가 아니라 정확한 Vim 조작을 해야 해결되는 압박이어야 한다.
- 브리핑은 짧게 쓴다. 한 화면에서 읽히지 않는 세계관 설명은 넣지 않는다.
- 실패 피드백은 세계관 농담보다 복구 힌트와 학습 의도 설명을 우선한다.
- 억까는 허용하지만, 항상 배운 command로 수습 가능해야 한다.
- 시나리오는 `target_state`, `optimal_keys`, `allowed_keys`, `constraints`를 바꾸지 않는다.
- command를 은폐하지 않는다. 플레이어는 자신이 어떤 Vim 개념을 연습하는지 알 수 있어야 한다.

## 권장 프레임

권장 배경은 “오래된 자동화 시스템이 고장 난 원격 시설”이다. 예시는 아래 범위에 둔다.

- 낡은 탐사 기지의 터미널 복구
- 격리된 서버실의 장애 대응
- 해저 관측소의 통신 로그 정리
- 폭풍권에 고립된 배포 콘솔 수습

배경은 바뀔 수 있지만 사건 단위는 아래 재료를 우선한다.

- 손상된 로그에서 위험 토큰 제거
- 잘못된 설정값 교체
- 정상 설정 줄 복제
- 구조 신호나 인증 토큰 일부 복사
- 특정 범위의 오염된 설정만 정리
- 실수한 변경을 `u`로 되돌리고 더 정확한 operator를 사용

## 좋은 억까 / 나쁜 억까

좋은 억까:

- “폭풍 때문에 입력 기회가 6번뿐입니다.”
- “구형 콘솔이라 화살표키가 경보를 울립니다.”
- “오염된 구역 표식 하나만 지워야 합니다.”
- “첫 줄은 조사 중이라 건드리면 안 됩니다.”
- “방금 과삭제했지만 `u`로 되돌리고 더 정확하게 다시 시도할 수 있습니다.”

나쁜 억까:

- 정답을 알아도 숨겨진 조건 때문에 실패한다.
- 평소 되던 Vim 동작이 설명 없이 다르게 동작한다.
- 한 번 틀리면 긴 구간을 처음부터 반복하게 한다.
- 체력, 배고픔, 인벤토리, NPC 선택지가 문항보다 중요해진다.
- 시나리오 장면을 먼저 만들고 나중에 Vim 문항을 끼워 넣는다.

## Command별 시나리오 매칭

| Command cluster | 학습 감각 | 잘 맞는 긴장 | 시나리오 예시 |
|-----------------|-----------|--------------|---------------|
| `delete-with-motion` | 불필요한 텍스트를 한 번에 제거 | 위험 플래그, 노이즈, 중복 줄 제거 | 산소 로그의 `leak` 토큰 삭제, 통신문 뒤 노이즈 `d$`로 제거 |
| `change-with-motion` | 삭제와 입력을 하나의 수정으로 묶기 | 잘못된 값 즉시 교체 | `valve closed`를 `valve open`으로 바꾸기, 줄 후반 설정 교체 |
| `yank-put-basic` | 안전한 값을 다시 치지 않고 재사용 | 정상 설정 복제, fallback 경로 구성 | 정상 라우팅 줄을 복사해 예비 라우트 생성 |
| `text-object-inner` | 커서 위치보다 구조를 대상으로 편집 | 따옴표/괄호 내부만 교체 | JSON 토큰 값, 함수 인자, env var 값 수정 |
| `search-basic` | 이동이 탐색으로 바뀌는 감각 | 원인 키워드 추적 | 로그에서 `ERROR` 또는 `timeout` 위치 찾기 |
| Ex substitute 응용 | 반복 오염을 범위 판단으로 정리 | 전체/범위 치환 선택 | 감염된 설정 블록 안에서만 이름 변경 |

## 중반 첫 Playpack 추천

다음 playable milestone은 operator grammar 입문이다.

| 순서 | 목표 command | 문항 감각 | 시나리오 톤 |
|------|--------------|-----------|-------------|
| 1 | `dw` | 위험한 단어 하나 삭제 | 냉각 로그의 잘못된 위험 플래그 제거 |
| 2 | `d$` | 현재 위치부터 줄 끝까지 폐기 | 구조 신호 뒤쪽 노이즈 제거 |
| 3 | `cw` | 단어 하나를 새 값으로 교체 | 잘못된 밸브 상태값 교체 |
| 4 | `c$` | 줄 후반 설정을 통째로 교체 | 라우팅 목적지 전체 교체 |
| 5 | `dd` | 오염된 한 줄 제거 | 중복된 위험 명령 줄 제거 |
| 6 | `u` 복습 + 더 정확한 operator | 과삭제 회복 후 정밀 수정 | 손이 미끄러진 뒤 되돌리고 올바른 범위 삭제 |

이 playpack은 `d`와 `c`를 먼저 익힌 뒤, 다음 playpack에서 `y/p`와 text object로 이어진다. 그렇게 해야 `ciw`, `diw`, `yiw`가 새 외계어가 아니라 “이미 배운 operator가 더 똑똑한 대상과 만난 것”으로 느껴진다.

## 제작 체크리스트

새 시나리오 아이디어는 아래 순서로 통과시킨다.

1. Command: 이번에 훈련할 Vim command cluster가 명확한가?
2. Exercise: 초기 상태, 목표 상태, optimal key trace, 필수/금지 입력이 검증 가능한가?
3. Scenario: 이 상황이 command의 필요성을 더 선명하게 만드는가?
4. Failure: 실패 이유가 `max_inputs`, `forbidden_keys`, `required_keys_missing`, 목표 미달 중 하나로 설명 가능한가?
5. Tone: 브리핑과 피드백이 짧고, 문항 목표를 가리지 않는가?

하나라도 실패하면 scenario로 승격하지 않는다.
